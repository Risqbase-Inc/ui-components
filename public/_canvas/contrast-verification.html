Audit findings

  1. FAIL: color.risk.medium + color.risk.high as text-on-white (2.15:1, 2.80:1). These tokens are documented as risk-band labels but the underlying primitives (amber-500, orange-500) cannot carry text on white. Used as fill behind dark text they're fine. Recommended fix: swap the risk-band semantic to alias darker stops of the new color.palette.risk.{600..800} ramp; flagged for v4.3.
  2. FAIL: color.band.very-low.text + color.band.high.text (white on emerald-500 = 2.54:1; white on orange-500 = 2.80:1). The dark-mode override (Phase 2.2) lifts the bg L* and flips text dark, clearing 7.8 / 7.4. Same lift-and-flip should apply to light mode in v4.3.
  3. FAIL: color.border.default + color.border.subtle (1.26, 1.09 vs white, target 3:1). Note: these are decorative borders per WCAG 1.4.11 carve-out (purely cosmetic dividers; informational borders are separate). Not an action item — but the audit doc flags the carve-out explicitly so future copywriting on the borders' role doesn't drift into "informational" framing.
  4. FAIL: color.citation.text-retracted on surface-low-conf (2.41:1). Retracted state is strikethrough + aria-disabled="true" per spec §7.12.7; low contrast is intentional ("do not engage"). Documented as an intentional carve-out, not a fix-it bug.

Method. WCAG 2.2 contrast ratio = (L1 + 0.05) / (L2 + 0.05), where L is relative luminance per WCAG 2.2 §1.4.3. sRGB transfer per IEC 61966-2-1. All values computed in JS at page load; refresh to recompute.
