UNRELEASED

Chromatic visual regression activated.

Closes v4.2.1 G5. Storybook 8.6 + Chromatic protect every component, every theme, every PR.

2026-05-14 · folds into v1.4.0 · spec v4.2.1

What this gives you

Every PR that touches src/, tokens/, or .storybook/ now produces a Chromatic build URL with side-by-side diffs across light, dark, HC, and print themes. Approving the build promotes the new pixels to baseline; rejecting it blocks merge. The dark / HC / print themes that just landed in BRIEF-401 are protected from silent regression.

What landed

  • 5 core components instrumented with co-located .stories.tsx files - Button, Badge, Header, Footer, SectionEyebrow.
  • 19 stories in four canonical shapes per component - Default · Variants · States/Composed/WithSibling · Gallery (print).
  • 62 captures in the initial baseline grid (stories × theme matrix, with print opt-in per Gallery).
  • PR-only workflow at .github/workflows/chromatic.yml - TurboSnap, autoAcceptChanges: main for first-time baselines, gated by the CHROMATIC_PROJECT_TOKEN repo secret.
  • Storybook 8.6 with addons essentials + themes + a11y + viewport. Pinned to ^8.6 majors per spec (Storybook 9 split addon-essentials, breaking the spec's config shape).

Why this matters now

v4.2.1 G5 ("visual regression baseline audit") was deferred to S5 of the implementation programme. With v1.4.0 about to publish to GitHub Packages and downstream consumers (marketing site this weekend, RALIA end June) starting to adopt, the regression-protection had to land before adoption ramps. Otherwise: silent theme drift in production with no diff to catch it.

BRIEF-401 - Substrate complete.

Audit + value-pass closes the v4.2.1 deferrals. Substrate complete; consumers can migrate.

2026-05-14 · package v1.1.1 → v1.4.0 · spec v4.2.1

Summary

The v4.2.1 patch (2026-05-11) shipped 241 enumerated tokens, the full 28-chart-type taxonomy, and the 84-term glossary - but explicitly deferred four substrate items: 38 chart-domain primitive hex values · dark + HC + print theme override values · scopes[] migration per v4.2.1 A7. BRIEF-401 closes all four, plus the implementation audit that surfaces four real AA-contrast failures (back-ported to v4.3 follow-ups).

What landed

  • 38 chart-domain primitive hex values - three sequential ramps (risk, financial, operational) × 10 stops + two diverging extensions × 4 stops. OKLCH-perceptually-equal-stepped; every stop AA-verified. See in the token tree →
  • Dark theme override values - 47 entries. Surfaces L\* 0.18 (never #000); brand brightens to indigo-500-equivalent #4768DB; risk bands lift-and-flip. AA-clean across the board.
  • HC (forced-colors) theme override values - CSS system keywords (Canvas, CanvasText, Highlight, …). Chart marks differentiate via shape / dash / pattern only.
  • Print theme + 4 pattern-fill SVG-data-URI assets - pure-white surfaces · 7-tier black ladder · pattern density tracks band severity inversely.
  • scopes[] migration patch - 131 entries (95 semantic + 36 component) carry the v4.2.1 A7 required scopes[] array.
  • Implementation audit - read-only PASS / FLAG / FAIL across token tree, AA contrast, components, Figma, consumers, visual regression. Read the audit →

Audit findings - back-ported to v4.3

Four real AA-contrast failures in the existing light-mode semantic layer surfaced during Phase 1. The dark-mode overrides already use a lift-and-flip pattern that fixes the analogous issue on dark canvases; the v4.3 patch will back-port the same to light:

Versioning convention

The package and the spec version separately. @risqbase-inc/ui-components is semver:

  • Major - breaking API change in a shipped component (renamed prop, removed variant, type-narrowing). Reviewed for migration cost across all three consumers before merge.
  • Minor - additive - new components, new tokens, new variants. BRIEF-401 was a minor (1.1.x → 1.2.0).
  • Patch - bug fix, contrast remediation, doc string fix. No new tokens; no new variants.

The spec versions semantic but separately. v4.x is the current ladder. A spec point release (e.g. v4.2 → v4.2.1) doesn't always trigger a package version bump - sometimes the spec lands ahead of code.
