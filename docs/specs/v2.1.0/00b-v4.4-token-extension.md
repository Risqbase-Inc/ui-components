## §1 The two gaps

### 1.1 Elevation stack

The audit-pack demo CSS (`audit-deliverable/styles/tokens-v4.2.1.css`) defines a three-token elevation stack and several demos/components consume `var(--shadow-floating)` against it. The shipped ui-components `tokens/` JSON has **no shadow tokens at all**. Components either inline `box-shadow` strings (fragile) or pull from Tailwind's defaults (inconsistent across components).

Demos referencing `--shadow-floating`: Marketing Demo B §4 (composition card), Demo C §2 (cockpit-shot wrapper), Demo D §2 (graph-shot wrapper), audit-deliverable redesigns (~6 instances). v2.1.0 specs referencing it: 01 ImpactGraph (MarketingImpactGraph chrome), 07 Header polish (dropdown panel), index.html (component cards).

**Action**: lift the three-token stack from the demo CSS into canonical tokens.

| Token | Value | Role |
|---|---|---|
| `--shadow-raised` | `0 1px 2px 0 rgb(0 0 0 / 0.04), 0 1px 1px -1px rgb(0 0 0 / 0.02)` | Cards at rest, subtle separation from surface |
| `--shadow-floating` | `0 4px 12px -2px rgb(0 0 0 / 0.06), 0 2px 4px -2px rgb(0 0 0 / 0.04)` | Floating panels (dropdowns, modals at rest, composition surfaces) |
| `--shadow-overlay` | `0 16px 40px -10px rgb(0 0 0 / 0.16), 0 6px 16px -6px rgb(0 0 0 / 0.08)` | Overlays, popovers, dialogs |

These values are **identical** to the audit-pack demo CSS — no design change, just promotion from demo-CSS-snippet to canonical-token-source. The values were tuned during the v4.2 / v4.3 cycle and have shipped against `risqbase-com` in the V2 demo HTMLs; they are de-facto canon.

### 1.2 Loading shimmer

Three v2.1.0 specs (01 ImpactGraph §2.3, 03 CompliancePostureStrip §5, 04 ClientGrid §5) reference `var(--color-skeleton-shimmer)` for skeleton/loading states. This token does not exist in the v2.0.0 release. The existing v2.0.0 `Skeleton` component (verified at `src/core/Skeleton/`) uses a hard-coded gradient inline. We are about to ship three more components that need skeletons; without a token, every skeleton bakes its own gradient.

| Token | Value | Role |
|---|---|---|
| `--color-skeleton-shimmer` | `linear-gradient(90deg, var(--color-surface-muted) 0%, var(--color-surface-subtle) 50%, var(--color-surface-muted) 100%)` | Skeleton placeholder fill; animated via the standard 1.4s linear infinite shimmer keyframe |

The value composes from `--color-surface-muted` and `--color-surface-subtle` — both shipped v4.3 primitives. **Zero new primitive colours.**

Standard shimmer keyframe (lift verbatim from existing Skeleton implementation):

```css
@keyframes risqbase-skeleton-shimmer {
  0%   { background-position: -200% 0; }
  100% { background-position:  200% 0; }
}
.risqbase-skeleton {
  background: var(--color-skeleton-shimmer);
  background-size: 200% 100%;
  animation: risqbase-skeleton-shimmer 1.4s linear infinite;
}
@media (prefers-reduced-motion: reduce) {
  .risqbase-skeleton { animation: none; }
}
```

The keyframe is a CSS utility, not a token. It lives in `dist/tokens.css` as a static block following the `:root` variable declarations.

## §2 What this brief is NOT

To avoid scope creep, the following are **out of scope** for v4.4:

- **`--color-finding-*` namespace** — these tokens appear in `audit-deliverable/styles/tokens-v4.3.css` (the demo build) but exist only to style the audit pack's own M-/I-/R- severity chips. They are not consumed by any ui-components production component. The v2.1.0 spec pack's `index.html` cover used them for one pill-styling element; this brief instructs that file to use the existing `--color-band-*` chain instead (see §3). No new tokens needed for ui-components.
- **`--color-band-very-low-bg` as a "semantic emerald"** — the token already exists in v4.3 (`tokens/semantic/color.json` line 222). Spec 04 ClientGrid REFINE 4.1 directs consumers to reach for it instead of `--color-palette-emerald-500`. No new token; semantic relabelling.
- **Dark / HC theme overrides** — the v4.3 dark + HC stubs (`tokens/themes/dark.json`, `tokens/themes/hc.json`) remain empty. v4.4 ships only the 4 light-mode additions above. Dark + HC values for shadows + shimmer are added in S5 alongside the rest of the theme work.

## §4 W3C token-format payload

To be appended to the ui-components `tokens/` JSON source. Format mirrors existing semantic-tier entries.

```jsonc
// tokens/semantic/elevation.json (NEW FILE)
{
  "shadow": {
    "raised": {
      "$value": "0 1px 2px 0 rgb(0 0 0 / 0.04), 0 1px 1px -1px rgb(0 0 0 / 0.02)",
      "$type": "shadow",
      "$description": "Cards at rest. Subtle separation from the surface plane.",
      "$extensions": {
        "com.risqbase.role": "semantic",
        "com.risqbase.figma": {
          "collection": "semantic",
          "mode": "light",
          "variable": "shadow/raised"
        }
      }
    },
    "floating": {
      "$value": "0 4px 12px -2px rgb(0 0 0 / 0.06), 0 2px 4px -2px rgb(0 0 0 / 0.04)",
      "$type": "shadow",
      "$description": "Floating panels — dropdowns, popovers, composition surfaces.",
      "$extensions": {
        "com.risqbase.role": "semantic",
        "com.risqbase.figma": {
          "collection": "semantic",
          "mode": "light",
          "variable": "shadow/floating"
        }
      }
    },
    "overlay": {
      "$value": "0 16px 40px -10px rgb(0 0 0 / 0.16), 0 6px 16px -6px rgb(0 0 0 / 0.08)",
      "$type": "shadow",
      "$description": "Overlays — modal dialogs, fullscreen lightboxes.",
      "$extensions": {
        "com.risqbase.role": "semantic",
        "com.risqbase.figma": {
          "collection": "semantic",
          "mode": "light",
          "variable": "shadow/overlay"
        }
      }
    }
  }
}
```

```jsonc
// tokens/semantic/loading.json (NEW FILE)
{
  "color": {
    "skeleton": {
      "shimmer": {
        "$value": "linear-gradient(90deg, {color.surface.muted} 0%, {color.surface.subtle} 50%, {color.surface.muted} 100%)",
        "$type": "color",
        "$description": "Gradient fill for skeleton placeholders. Composed from surface muted+subtle; animated via the standard 1.4s shimmer keyframe.",
        "$extensions": {
          "com.risqbase.role": "semantic",
          "com.risqbase.figma": {
            "collection": "semantic",
            "mode": "light",
            "variable": "color/skeleton/shimmer"
          }
        }
      }

  }
}
```

Style Dictionary handles the `{color.surface.muted}` references and emits the resolved CSS variable in `dist/tokens.css`.

## §7 Lesson encoded

This brief exists because the v2.1.0 spec pack assumed tokens lived in `tokens/` JSON that only lived in `audit-deliverable/styles/`. The demo-vs-source drift is a recurring failure mode (cf. β P5 reviews). Two ways to harden:

1. **Spec authoring**: every `var(--*)` reference in a v2.x spec must be verified against `dist/tokens.css`, not against demo CSS. Add a token-existence check to the spec-pack lint (Claude Design or Claude Code on Web automation).
2. **Demo CSS**: `audit-deliverable/styles/tokens-v4.x.css` files become **direct re-exports** of `dist/tokens.css`, not hand-authored extensions. Eliminate the drift surface.

Both are v4.5+ work — not in scope for this brief, but worth Elena G4 + orchestrator alignment before the next spec pack.
