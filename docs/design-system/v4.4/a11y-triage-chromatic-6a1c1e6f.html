3Root-cause clusters & fix specs

Ranked by violation mass. Confidence reflects what counts alone can prove; every cluster lists its one-story confirmation probe — pull the axe rule list for that single story and the hypothesis is confirmed or killed.

C1 · Dark faint-text tier fails 4.5:1 confirm via ClientGrid / All Mode 12 Clients / dark

est. 600–900 of 1,257 dark

The B1 derivation set dark text.subtle → stone-300 (safe), but component-tier tokens that resolve to fainter stones were likely never re-pointed. The killer pair: Aa500 stone-500 on stone-900 ≈ 3.5:1 — fails normal-text 4.5:1; on stone-800 ≈ 3.05:1, worse. Every metadata line, timestamp, caption, helper text, and tertiary label using that tier fails per element — which is exactly why text-dense stories (client cards, posture strips, footers) post the huge counts.

Token-level, not component-level: add a dark override for every component token resolving to stone-500/stone-400 in a text role — re-point to stone-400 (≈6.8:1 on stone-900) or stone-300. Then extend verify-contrast from the 8 curated pairs to every token whose name matches a text role, asserted per theme in CI (the gap this report exposes). Components needing zero code changes if the chain is clean.

C2 · Inverse-surface chrome not double-flipped confirm via Footer / Default / dark

Footer 182 + Header share ≈ 300

Footer rides surface.inverse (stone-900 + light type in light theme). B1 flips surface.inverse → white in dark — but if text.on-inverse-subtle (and nav-link/legal-text component tokens) kept their light-theme values (light stones), they're now light-on-white. Footer's 182 dark violations across every story, with light at 0, is the textbook signature. Header's dark dropdown panels/nav states likely share a variant of this.

Derive the full on-inverse chain for dark (text, subtle, dividers, hover states) and add contrastPair entries pinned to surface.inverse so the pair set tracks the flip automatically. Acceptance: Footer Default/Mobile dark = 0.

C3 · Tinted-subtle intent/band surfaces in dark confirm via Callout / Intents / dark

Callout 27 · Badge 27 · CitationChip 42 · Toast 16 ≈ 112

Intent surfaces (callout.*, citation.surface-*, toast intents, badge band-subtle text) pair a tinted background with intent-colored text. If dark kept light-theme text values on darkened tints — e.g. amber-700-class text on a dark amber wash — ratios collapse. The "Intents"/"Gallery"/"All Variants" stories failing at 8–11 (≈ one per intent/variant shown) while single-variant stories fail at 1–2 fits exactly.

Re-derive intent/band text + icon tokens for dark (lighten text toward the 200–300 weights, keep hue), mirroring the B1 band rule already applied to band.*.bg. Add one contrastPair per intent per theme.

C4 · ClientGrid fails in light too — pre-token component own remediation

25 unique light + 369 dark

The only component failing at scale in light (5 per multi-client story, ~1 per card row) — meaning raw values that were never on the token chain (and are R9-baselined, so the scanner ignores them until the lines drift). Prime suspect: stone-400-class metadata on white ≈ 2.36:1. Its dark numbers then compound C1 on top.

Migrate ClientGrid (and CompliancePostureStrip, same family) onto semantic text tokens outright — that fixes light and dark in one move and burns down baselined R9 debt. This is the single highest-leverage component fix in the report: 695 of 1,344 violations (52%).

C5 · Gallery-story landmark duplicates moderate · story bug, not component bug

exactly 3 × (light+hc+print) on Header/Footer galleries

Footer "Print Gallery" and Header "Gallery"/"Variants" render multiple instances per story → duplicate banner/contentinfo landmarks. Axe's three landmark rules (landmark-no-duplicate-banner, -contentinfo, landmark-unique) = the suspiciously exact "3" in light, hc and print. ClientScopeBanner "Stack" (1) is the same idea with a region/complementary role.

In the gallery stories only: render duplicate instances inside containers with role="presentation" (or pass an existing as="div"-style prop), keeping one canonical landmark per story. No component change.

C6 · Mode-independent structurals + not-run rows

ImpactGraph 3 · MarketingImpactGraph 6 · 6 not-run

ImpactGraph "Interactive" = 1 in all three modes → not contrast; a structural rule (nested-interactive, focusable inside role="img", or missing accessible name). MarketingImpactGraph fails light/hc but is clean in dark — the inverse pattern: a dark-substrate marketing surface whose caption/legend (stone-400-class on white chrome) only fails against the light page. Not-run ≠ pass: Modal/Drawer/Sheet "Interactive" and TelemetryBeacon never executed axe — mark TelemetryBeacon as intentionally excluded (it renders nothing) and fix the Modal Interactive story so the dialog is mounted at snapshot time.

One-line fixes once rule IDs are pulled; the Modal story needs its play-function to await the open state before the a11y pass.

4Execution protocol

  1. Confirm with five probes (½ hour). Pull axe rule lists — Chromatic UI a11y tab, or locally via axe-core against Storybook with the theme globals — for: ClientGrid All Mode 12 Clients (dark + light), Footer Default (dark), Callout Intents (dark), StreamingText Default (dark), Header Gallery (light). These five confirm or kill C1–C5 and reveal C6's rule IDs.
  2. Fix at the token layer first (C1 → C2 → C3): dark overrides for faint-text tiers, the on-inverse chain, and intent/band text. Expect roughly 1,100 of the 1,257 dark violations to collapse from token edits alone, zero component code.
  3. ClientGrid / CompliancePostureStrip token migration (C4) — fixes the entire light column and their dark mass together.
  4. Story-level fixes (C5, C6): gallery landmark wrappers, the ImpactGraph structural, the Modal Interactive mount timing.
  5. Re-run Chromatic once, then accept baselines once. Single re-baseline pass after the fixes — accepting today's build first would mean re-accepting everything again after the fixes anyway.
  6. Permanent gates: (a) extend verify-contrast to all text-role tokens per theme; (b) treat Chromatic a11y deltas as review-blocking for dark-mode PRs; (c) ledger rows A5.6/B5.3/F.4 only close after the re-run is green-and-accepted.

Gate-order change: earlier guidance was "review build 101 and accept the dark baselines". This report supersedes that — hold acceptance until steps 1–4 land. Light-theme zero-drift is unaffected (A5.5's mechanical proof stands); it's the dark/HC baseline acceptance that should wait.
