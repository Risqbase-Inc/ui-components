## Button border-radius — brand-baseline reconciliation

**Status:** Open — needs CEO + Elena adjudication for v4.2.1.

The v4.2 spec does not explicitly mandate a border-radius for
`<Button>`. The closest signals point in two different directions:

1. **`tokens/primitive/dimension.json` line 15** describes the
  `radius.lg` primitive (12px) as `"Large radius — buttons"`. This
  is the canonical statement on button rounding in the v4.2 token
  system. v1.x consumers ship `rounded-xl` (= `radius.lg` = 12px).
  S1's `<Button>` keeps `rounded-xl` and `s1/cleanup-3-design` adds
  the missing `dimension.radius.button.default` role token aliasing
  `{dimension.radius.lg}` so spec §15.4 is honoured.
2. **`GOV-DS-2026-01` brand baseline** specifies `rounded-full` for
  buttons across RisqBase product surfaces (CTAs, primary actions).
  Skip links in `spec.md` line 198 use the "indigo brand pill style"
  in keeping with the brand baseline.

The two anchors disagree. v4.2 spec line 28 ("This is the canonical
world-class baseline. All subsequent revisions are deltas to v4.2.")
makes the spec the system-level contract, but GOV-DS-2026-01
predates v4.2 and is the brand baseline that the marketing site,
RALIA, and Cortex consume directly.

**Decision in `s1/cleanup-3-design`:** keep `rounded-xl` (no pixel
change to v1.x consumers; honour the primitive token's explicit
`"Large radius — buttons"` description). Add the
`dimension.radius.button.default` role token (alias to
`{dimension.radius.lg}`) so `<Button>` consumes a role token per
spec §15.4 rather than a Tailwind utility magic constant.

**v4.2.1 question for the CEO + Elena:** is GOV-DS-2026-01's
`rounded-full` baseline still the brand intent for buttons, or has
v4.2's adoption of `rounded-xl` (via the `radius.lg` primitive) made
the pill standard obsolete for buttons (and retained only for
badges and skip links)? Whichever way it lands, the v4.2.1 spec
patch should write it down explicitly in §15.6 (token enumeration)
or §13 (Components retained from v3.1) so future agents do not
re-discover this tension. See
`docs/design-system/v4.2/notes/button-radius-decision.md` for the
S1 rationale in full.
