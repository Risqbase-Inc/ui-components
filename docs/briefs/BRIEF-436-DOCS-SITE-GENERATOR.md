# BRIEF-436 — design.risqbase.com docs-site generator

**Status:** IMPLEMENTED 2026-05-20 · CEO Path B 2026-05-20 ratified
**Owner:** Jordan Park (G6) · PR train author
**Reviewers:** Alex Chen (G1), Marcus Chen (G2)
**Closes:** docs-site v1.4.0 staleness (version pill, hero stats, missing v2.0.0/v4.3 entries)

## §1 Problem

`design.risqbase.com` was hand-maintained — every `public/*.html` carried the version pill `v1.4.0`, hero stats hardcoded to `5 shipped components` + `4 themes (light · dark · HC · print)`, and a changelog frozen at the v1.4.0 cut. The release-please bot ships v2.0.0 + v4.3 to `CHANGELOG.md` and bumps `package.json`, but nothing propagates to `public/`. Each release leaves the docs site lying about reality.

A point-in-time hand-patch closes the immediate gap; a generator closes the class.

## §2 Decision

CEO Path B (2026-05-20): replace hand-maintained HTML with a release-driven generator. Templates carry the durable layout/copy; the generator injects the volatile bits (version, counts, changelog) from the existing sources of truth on every CI run.

## §7 Template placeholder syntax

Mustache-style `{{PLACEHOLDER}}` — string-replace via Node, no runtime templating engine. Unknown placeholders fail the build loudly (no silent empties). Two-pass substitution allows a placeholder value to itself contain another placeholder; the generator caps at 2 passes (recursive substitution = footgun).

Available placeholders:

```
{{PKG_VERSION}} {{SPEC_VERSION}}
{{TOKEN_COUNT}} {{COMPONENT_COUNT}} {{CHART_TYPE_COUNT}} {{THEME_COUNT}}
{{THEME_LIST}} {{CHART_TYPES_LIST}}
{{COMPONENT_GRID_CORE}} {{COMPONENT_GRID_AI}} {{COMPONENT_GRID_VIZ}} {{COMPONENT_GRID_MARKETING}}
{{COMPONENT_LIST_CORE}} {{COMPONENT_LIST_AI}} {{COMPONENT_LIST_VIZ}} {{COMPONENT_LIST_MARKETING}}
{{CHANGELOG_ENTRIES}} {{CHANGELOG_NAV}}
```

Extending: add the extractor to `lib/extractors.mjs`, the renderer to `lib/render.mjs`, the placeholder to the `context` object in `index.mjs`, reference it in the template. Then `npm run docs:build` and inspect the diff.

## §8 Rejected alternatives

| Approach | Why rejected |
|---|---|
| Static-site generator (Astro, Eleventy, Vite-SSG) | Adds 100s of deps for a 6-page site; Vercel-deploy story stays single-source on `public/` |
| Inline-JS rendering on page load | SEO hit; Vercel edge would have to ship JSON sidecars; defeats the "static HTML in `public/`" Vercel auto-deploy contract |
| Migrate to MDX / Markdown | Templates already pixel-perfect; rewriting in MDX risks visual regression for zero new capability |
| `nunjucks` / `handlebars` runtime | One more dep to maintain; our placeholder set is narrow + stable; string-replace is auditable in 20 lines |

## §9 Operations

- **Adding a new release:** release-please writes to `CHANGELOG.md` and bumps `package.json#version` → next `main` push triggers `docs-publish.yml` → regen + commit `public/*.html` → Vercel auto-deploys.
- **Adding a component:** create `src/<domain>/NewThing/index.tsx` → next push regens components.html grid + index.html hero count automatically.
- **Editing copy:** change `tools/docs-build/templates/<page>.html` → run `npm run docs:build` locally → commit both template + generated public/.

## §10 Drift discipline

- Pre-commit (future): consider a Husky hook running `docs:check` to remind devs.
- CI gate today: `npm run docs:check` blocks merge if drift detected.
- Post-merge safety net: `docs-publish.yml` regenerates on `main` push, commits back with `[skip ci]`.

## §12 Why this is the enduring fix

The doc-site staleness wasn't a one-time bug — it's a coordination problem. Every release introduces drift between `package.json`, `CHANGELOG.md`, `src/`, and `public/`. Hand-syncing has a half-life. Generating from the sources puts the coordination on the machine, where it belongs.
