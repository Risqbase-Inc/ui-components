## §2 Cross-cutting findings

### NIT-1 — `Math.random()` for `figureId` in ImpactGraph (hydration regression risk)

**Severity**: IMPROVEMENT
**File**: `src/data-viz/ImpactGraph/ImpactGraph.tsx:104`

```ts
const figureId = useMemo(() => `ig-${Math.random().toString(36).slice(2, 9)}`, [])
```

Every other v2.1.0 component correctly uses React `useId()`:
- `CompliancePostureStrip/index.tsx:161` → `useId()`
- `ClientGrid/index.tsx:74,75,237` → `useId()` (×3)
- `HeroVideo/index.tsx:203` → `useId()`
- `Header/NavDropdown.tsx:47` → `useId()`

`Math.random()` produces a different value on server vs client, triggering React hydration warning when ImpactGraph renders in SSR (Marketing-site path). The IDs reference `<title>` and `<desc>` which carry the SR accessible-name — so a hydration swap can briefly mismatch the `aria-labelledby` target during the React reconciliation window.

**Fix** (4-line PR): replace with `useId()`, then `const figureId = \`ig-${reactId.replace(/:/g, '')}\``. The colon-strip is needed because `useId()` returns ids of form `:r5:` which are invalid in SVG IDREFs without quoting.

### NIT-2 — `figcaption` is rendered but never referenced

**Severity**: OBSERVATION
**File**: `src/data-viz/ImpactGraph/ImpactGraph.tsx:431`

The SR-only `<figcaption id="${figureId}-caption">` enumerating affected entities is rendered as a descendant of `<figure>` but no `aria-describedby` points to it (`<figure>` and `<svg>` both `describedby={figureId}-desc` — the `<desc>` inside the SVG, not the figcaption). NVDA and VoiceOver *will* announce `<figcaption>` content as part of `<figure>` traversal in most cases, so this is not a P0 a11y break. But the accessibility.md §1.2 promises a "navigable representation of the graph itself" via this list, and per WCAG 1.3.1 we should not rely on heuristic SR behaviour. Suggest either (a) join the desc + figcaption into a single describedby target via `aria-describedby={"${figureId}-desc ${figureId}-caption"}`, or (b) document the rely-on-figure-traversal as the intentional pattern in accessibility.md and verify in the §5.2 SR walkthrough.

### NIT-3 — No test runner in `ui-components`

**Severity**: IMPROVEMENT (recommend separate brief)
**Repo-wide**

`package.json` declares no `vitest`, `jest`, or `@storybook/test-runner` dependency, and no `test` npm script. The 5 Header play functions use `@storybook/test`'s `expect/userEvent/within/waitFor` APIs at story-render time inside Storybook — they fire when you open the story locally or under Chromatic, but **nothing in CI runs `vitest --workspace storybook` or `test-storybook` to fail a build when a play assertion throws**. They are documentation + manual-QA scaffolding right now, not regression gates.

**Concrete impact**: a future PR could break ESC-closes-and-restores-focus, Header.stories.tsx would still build, Chromatic visual diff would NOT catch it (focus ring on trigger looks identical whether or not focus is actually there), and the bug ships.

**Recommendation**: file a follow-up brief, **BRIEF-ui-test-runner** (~6h). Scope:
1. Add `@storybook/test-runner` (`npm i -D`).
2. Add `npm run test:stories` → `test-storybook --url http://127.0.0.1:6006`.
3. Wire CI: parallel job that runs `npm run build:storybook && http-server storybook-static & test-storybook`.
4. Acceptance: a `play:` function whose `expect(...)` fails causes CI red.

This is BRIEF-level work (not a NIT to absorb inline) because it touches CI matrix + Storybook config + likely needs flake discipline (play-function timeouts) and the test-runner has its own setup cost.

### NIT-4 — Chromatic acceptance discipline not codified

**Severity**: IMPROVEMENT (recommend short policy doc, not a brief)
**Repo-wide**

v2.1.0 added ~100+ new Chromatic baselines (12 ArcDecoration + 12 ImpactGraph + 4 MarketingImpactGraph + 10 HeroVideo + 10 CompliancePostureStrip + 12 ClientGrid + 8 CitationChip + several Header polish). With Chromatic as the de facto test framework, the **accept-baseline discipline** is the actual quality gate. No written policy currently governs:

- Who can accept a baseline change (just the PR author? requires another reviewer?)
- What counts as an *expected* visual change (token change → accept all; component refactor → spot-check by hand) vs an *unexpected* one (catch a bug)
- Threshold tolerance defaults
- What to do when Chromatic shows a diff and the PR description says nothing about visual changes

**Recommendation**: a one-page `docs/contributing/chromatic-acceptance.md` (~1h). Mandatory clauses to include based on my `pattern_test_masking_zero_baseline.md` learning:
1. Every accepted baseline change requires a one-line PR-description rationale ("expected — token-name rename"; "expected — Spec 07 §3.2 dropdown chevron added"; etc.)
2. Two-reviewer accept on Marketing surfaces (brand drift = product damage)
3. Pixel-diff threshold: 0.1% (not Chromatic default 0.063 — see ~/.claude/agents/sarah-mitchell-memory pattern on flake-reduction)
4. Reject + investigate if Chromatic diff appears on a PR that didn't touch the component's source or tokens (drift signal)
5. Mobile + desktop baselines required for every responsive component (already enforced de facto; codify)

### NIT-5 — Interactive-path play functions absent on ImpactGraph + CompliancePostureStrip + ClientGrid

**Severity**: IMPROVEMENT (recommend follow-up PR once test-runner wired)
**Files**: `ImpactGraph.stories.tsx:177` (Interactive), `CompliancePostureStrip.stories.tsx:95` (Interactive), `ClientGrid.stories.tsx:132` (InteractiveCards), `ClientGrid.stories.tsx:140` (InteractivePill)

Four interactive stories exist with named handlers (`onEntityClick`, `onAlertClick`, `onClick`, `onClientClick`). The story descriptions explicitly claim keyboard behaviour ("Tab into the SVG... Enter or Space"; "Focus ring is the platform-canon 2px"; "`event.stopPropagation` so the inner click does not bubble"). **None of these claims is asserted at story-render time.** Header proves the pattern works: 5 tight play functions, each `await step('description', async () => { ... })`, each with `await expect(...).toHaveAttribute(...)` assertions. Mirror the Header pattern on:

1. ImpactGraph `Interactive`: focus first node → assert focus ring class → press Enter → assert `onEntityClick` fired with correct entity
2. CompliancePostureStrip `Interactive`: focus first cell → assert `<button>` role → press Enter → assert `onClick` fired
3. ClientGrid `InteractivePill`: click card area → assert click did NOT fire `onAlertClick` (pill `stopPropagation` claim); click pill → assert `onAlertClick` fired exactly once
4. ClientGrid mutex warn-path: pass both `onClientClick` and `onAlertClick` → assert dev-mode `console.warn` fires + card-click path is the active one

These tests don't gate the v2.1.0 release (we already shipped). They're follow-up PR 1 of a hardening sweep that should land once NIT-3 (test runner) is wired.

### NIT-6 — HeroVideo state-machine has zero behavioural coverage

**Severity**: IMPROVEMENT (follow-up PR)
**File**: `src/marketing/HeroVideo/HeroVideo.stories.tsx`

577-line component with a 5-state machine, IntersectionObserver gate, 5s autoplay-retry timer, reduced-motion gate, VTT captions, prefers-reduced-motion media listener. The 10 stories cover the *visual rendering* of each state, but no story asserts the *transitions*. Plausible mutations not caught today:
- Flip `if (state === 'idle')` to `if (state !== 'idle')` in IO callback → idle state never advances → no Chromatic story fails (PosterOnly story still renders identical poster)
- Off-by-one on the 5s retry: `setTimeout(..., 5000)` → `setTimeout(..., 500)` → retry fires 10× too aggressive → no test catches
- Drop the `prefers-reduced-motion` media listener → motion plays on reduce → only ReducedMotionFallback story would catch, and only in Chromatic's reduced-motion mode (not in CI default)

This compounds with NIT-3 (no test runner). Once a runner exists, add 3-4 play functions: (i) idle→playing on IO fire (mock IO via `window.IntersectionObserver` in story setup); (ii) playing→errored after 5s with broken source; (iii) reduced-motion blocks autoplay; (iv) captions-default-on adds `<track default>` to the rendered video element.
