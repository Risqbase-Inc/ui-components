# G4 · v2.1.0 post-merge visual + a11y audit

## §0 Headline

**PASS_WITH_NITS.** v2.1.0 ships with strong design-system discipline. 6/6
components route every visual surface through `var(--color-*)` / `var(--dimension-*)`
/ `var(--shadow-*)` token chains; zero prohibited `blue-*` / `gray-900` /
raw `--color-palette-*` violations across the merged surface area (verified
via grep, see §6). Accessibility posture meets WCAG 2.2 AA on every
component: focus-visible rings with 3:1+ contrast, prefers-reduced-motion
respected on every animation, hit-targets ≥24×24 CSS px on interactive
controls, structured SR equivalents on every non-trivial graphic.

Per the CEO 2026-05-20 admin-merge directive, findings are filed as
**follow-up tickets** rather than revert proposals. Severity floor for any
post-merge revert request is "safety-critical visual regression that
materially breaks the marketing surface" — none of the 11 findings below
clear that bar.

The two cross-cutting concerns flagged in the brief (Liam's pixel-fidelity
flag on MarketingImpactGraph; Alex's ResponsiveBreakpoints viewport flag
on ClientGrid) are both **acceptable as-shipped** with documented
follow-up paths; rationale in §3.

### 2.2 MarketingImpactGraph (wrapper)

**Files audited:**
- `src/data-viz/ImpactGraph/MarketingImpactGraph.tsx`
- `src/data-viz/ImpactGraph/fixtures/edpb-04-2026.ts`
- `src/data-viz/ImpactGraph/fixtures/marketing-categories.ts`
- `src/data-viz/ImpactGraph/MarketingImpactGraph.stories.tsx` (4 stories)

**Liam's pixel-fidelity flag — adjudicated:**

I compared the shipped layout engine against Demo D source positions
(line 270-344 of `Marketing Demo D - HorizonIris Impact Graph.html`).
Demo D hand-places 14 nodes at hard-coded coordinates (cx=380/470/580/680/800
for DPIAs along the top arc, etc). The shipped MarketingImpactGraph
generates positions via category-sector polar layout + deterministic
hash jitter from `entity.id`. **The output is NOT pixel-identical to
Demo D.** Difference is:

- Demo D positions are visually art-directed (varying radial distance
  for visual rhythm; e.g. `dpia-biometric-access` at r≈200 sits closer
  to the centre than `dpia-cctv-acme-lobby` at r≈170).
- Shipped output places all DPIA nodes at the same baseRadial+jitter
  band per sector, which produces a slightly more uniform ring than
  Demo D's hand-tuned spread.

**Verdict: ACCEPTABLE.** Rationale:

1. The visual *signature* — alert at centre, categorical sectors, edge
   labels, cascade dashed line, Iris attribution badge top-left, legend
   strip below — is preserved. A marketing visitor scanning the page in
   3 seconds gets the same gestalt.
2. The deterministic-layout approach has architectural value Demo D
   lacks: it scales to ManyEntities (30+) without hand-tuning, and any
   future fixture (BRIEF-XXX next regulatory alert) gets the same
   visual treatment automatically. Pixel-identical layout would require
   per-fixture coordinate maps, which is exactly the maintenance burden
   the engine was designed to escape.
3. The hash-jitter is *visually distinct enough* from a perfect ring
   that the layout reads as "organic graph", not "pie chart with extra
   steps" — which was the chief visual risk of going procedural.

**Follow-up path (NOT a blocker):**

| # | Severity | Finding | Suggested follow-up |
|---|---|---|---|
| MIG-1 | IMPROVEMENT | Pixel-fidelity drift vs Demo D is acceptable but ungated — a future ImpactGraph layout refactor could shift the marketing canvas without warning. | Add a Chromatic story `MarketingImpactGraph/DemoDFidelitySnapshot` that captures the current canvas as the design canon; any future drift requires explicit Chromatic re-approval. Effort: 30 min (story + baseline). Owner: ImpactGraph maintainer. |
| MIG-2 | IMPROVEMENT | If a future fixture *requires* hand-placed positions (e.g. for an art-directed RALIA-app hero), the engine has no escape hatch. | Optional `positionOverrides?: Record<entity.id, {x:number, y:number}>` prop on `ImpactGraph` that, when present, bypasses `layoutEntities` for matching IDs. Effort: 1 hr. Defer until 2nd fixture lands per "ship platform standards immediately" judgment — but file as a follow-up brief now so we don't lose it. |

**Liam's REFINE 1.2 `-border` vs Demo D `-bg` divergence — adjudicated:**

Shipped code uses `var(--color-band-{severity}-border)` for edge strokes
(`layout.ts:29-34`). Demo D uses `var(--color-band-{severity}-bg)` (HTML
source line 410). The `-border` chain is the spec-canonical decision
from REFINE 1.2 (BRIEF-401 cleanup that established `-bg` = fill,
`-border` = stroke). The Demo predates that token discipline.

**Verdict: ACCEPTABLE — spec correctly overrides the Demo.** The visual
difference at -bg vs -border is ~5% in saturation on the rendered
stroke; well within the band identity. No follow-up.

### 3.2 Reduced-motion discipline (PASS)

Every component with animation respects `prefers-reduced-motion: reduce`:

- ImpactGraph centre-ring pulse: `@media (prefers-reduced-motion: no-preference)` gate
