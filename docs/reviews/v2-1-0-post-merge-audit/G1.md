# G1 Architecture Audit — v2.1.0 (post-merge)

## Verdict

**CONCERNS** — Architecture is broadly sound. APIs are clean, the client/server boundary is correctly drawn, the marketing/primitives namespace promotions are well-documented, and the v4.4 token extension is a textbook "promote demo-CSS-only values to source-of-truth" move. No revert candidates; no safety-critical bugs.

Two real bugs (`ImpactGraph` SSR-unsafe id; `ClientGrid` unguarded `process.env`) plus a handful of consistency NITs warrant follow-up PRs. None block any consumer today; all are cheap.

The single cross-cutting concern worth a brief is the **v4.4 Style Dictionary alias-resolution workaround** Priya flagged — leaving the inline `var()` fallback in place creates a 2-token-system surface that will bite us at v4.5+ when more semantic aliases compose.

## Scope check

Two namespace topology changes shipped without a written ADR. I'd backfill ADRs for both — they're irreversible-ish (consumers' import paths now depend on them) and worth a paragraph of context:

- `src/marketing/` — new namespace, new tsup entry, new package.json subpath export, new root-barrel re-export. Convention: "marketing-tier compositions (lazy-loaded by default, captions required on media, no live data binding)." Header comment on `src/marketing/index.ts` captures the rationale well; an ADR formalises it.
- `src/primitives/` — flipped from "INTERNAL ONLY" (v2.0.0 intent) to "public + internal mixed namespace". Header comment on `src/primitives/index.ts` captures the rationale well; same — ADR.

Both are documented in-tree (good) but not in `docs/adrs/` (where future engineers go to ask "why this shape?").

## Cross-cutting concerns

### 1. v4.4 Style Dictionary alias-resolution — follow-up brief warranted

Priya flagged that the tools/tokens-build pipeline can't resolve composite-token aliases cleanly, requiring inline `var()` workarounds in the W3C JSON source. Two of the four v4.4 tokens (`shadow.*` and `color.skeleton.shimmer`) leaned on workarounds. **At v4.4 this is tolerable** — three elevation shadows are leaf primitives anyway, and the shimmer's `linear-gradient(...{color.surface.muted}...)` is the only true composite.

But v4.5+ wants heatmap, area, choropleth, metric-card — those will compose deeply (semantic palettes that derive from chart-cat-* primitives, motion-token composites, etc.). The workaround approach won't scale. **File a follow-up brief**: "Style Dictionary v4 composite-token resolution — pipeline upgrade for v4.5". Owner: Priya. Effort: probably 4-6h to evaluate `style-dictionary@4.x` resolver hooks vs. switching to `tokens-studio/sd-transforms`.

### 2. ADRs missing for `marketing/` + `primitives/` namespace promotions

Both decisions are well-documented in source-code header comments. Neither has an ADR. Both are irreversible-ish (consumer import paths now depend on them). **Backfill two short ADRs**:

- `docs/adrs/ADR-001-marketing-namespace.md` — why a Layer-2 namespace separate from core, lazy-load convention, captions-required contract.
- `docs/adrs/ADR-002-primitives-public-internal-coexistence.md` — why ArcDecoration flipped the namespace from internal-only, what stays internal (Slot etc.), G4 review required to promote.

Both are 1-2h tasks. Not blocking.

### 3. `process.env.NODE_ENV` guard convention is split

`ImpactGraph` + `src/index.ts` use the guarded form (`typeof process !== 'undefined' && ...`). `ClientGrid` uses the unguarded form. Pick one and ESLint-enforce. Worth a tiny `eslint-plugin-local` rule or `no-restricted-syntax` config. **Sweep ticket**.

### 4. Tailwind arbitrary-classes vs. computed `style={}` — convention split

`CompliancePostureStrip` + `ClientGrid` use Tailwind arbitrary-class syntax (`bg-[var(--color-skeleton-shimmer)]`); `ImpactGraph` + `HeroVideo` + `ArcDecoration` use inline `style={{ background: 'var(...)' }}`. Both work. Tailwind-arbitrary requires the consumer's content-scan to cover the dist output; inline-style is self-contained but loses Tailwind's IDE/colour-on-hover affordances. Worth picking one for v4.5 and documenting; for v2.1.0 it's a watch-item only.

## Forward-compat assessment

Will v4.5+ work be backward-compat with these APIs? **Yes, with two caveats:**

1. **`CategoryPalette.color: string` (free-form var() reference)** — if v4.5 introduces a `palette.*` semantic-token chain with stronger typing, we'll want `color: PaletteToken` (a branded type). That's an additive change (we can widen `string` to `string | PaletteToken` first, then deprecate `string`). Path is clean.
2. **`HeroVideoBeat.index: 0 | 1 | 2 | 3`** — explicit literal type pins to 4 beats forever. If we ever want a 6-beat hero variant, that's a breaking change. Worth weighing now: do we commit to "4 beats is the contract" forever, or widen to `number` and validate at runtime? I lean keep-it-pinned (the 2×2 fallback grid is structurally 4); document as such in `accessibility.md`.
